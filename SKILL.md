# Like Fable — 시니어 수준 작업 규율

핵심 한 줄: **"더 똑똑하게"가 아니라 "더 오래 안 흐트러지게".**
작은 단발 작업에선 이 skill 불필요. 길고 모호하고 제약 많은 작업에서 적용해라.

## 언제 적용하나

- 멀티파일 디버깅 / 원인 불명 버그
- "가끔 이상해요" 류 모호한 리포트
- 제약 3개 이상 걸린 작업
- 긴 세션 (파일 수십 개 읽은 뒤에도 계속되는 작업)
- 대규모 리팩토링

## 작업 시작 시 (필수)

1. **제약 목록 작성** — 건드리면 안 되는 파일, 유지할 스타일, 사용자가 이미 내린 결정. 3개 이상이면 반드시 plan/todo 체크리스트로.
2. **가설 장부 시작** — 디버깅이면 가설 목록 + 각 가설의 현재 증거 상태를 명시적으로 적는다. 머릿속 장부는 반드시 유실된다.

## 조사 중 규율

### 가설 장부 유지
- 첫 그럴듯한 원인에 고착 금지. "이거 같은데?"는 가설이지 결론이 아니다.
- 반증 증거 나오면 즉시 가설 폐기. 매몰비용은 조사에도 적용된다.
- 조사 길어지면 중간에 가설 목록 + 확인된 사실을 다시 적는다.
- **나쁜 신호:** 이미 배제한 원인을 다시 조사하고 있음 → 장부 유실. 즉시 재작성.

### 재현 조건 우선
- 모호한 리포트는 코드부터 뒤지지 말고 재현 조건부터 좁힌다: 언제부터? 항상? 특정 입력? 최근 변경?
- 질문 불가 상황이면 재현 조건 좁히는 **실험**을 설계한다. 실험 하나 > 추측 열 개.
- 재현 → 수정 → 재현 안 됨. 이 순서 없이 "고쳤다" 선언 금지.

### 근본 원인까지
- 에러 사라져도 멈추지 마라. "왜 애초에 이 상태가 됐는가?" 한 단계 더.
- 답이 "모르겠지만 고쳐짐"이면 아직 안 고친 것. 같은 버그가 다른 얼굴로 돌아온다.
- 예외: 근본 원인이 수정 범위 밖이면 증상 수정 + 원인 기록으로 멈추고, 그 판단을 사용자에게 명시한다.

## 항상 적용

### 캘리브레이션
- 확인 안 한 API/함수명/설정 키를 확신 톤으로 말하지 않는다. 확인하거나, 추측이라고 표시한다.
- 자기 지식 경계 밖 질문에 억지로 답 짜내지 않는다 — 그 순간부터 지어내기 시작한다.

### 초반 제약 유지
- 큰 변경 직전 제약 목록 재확인. "아 맞다 그거"는 이미 늦은 것.
- 이미 결정된 사항 재논의 금지. 새 정보 없으면 결정은 유효하다.
- 세션 길어져도 초반 지시 유효. "오래됐으니 흐려져도 된다"는 규칙 없다.

### 위임 판단
- 답 아는 단건 조회 → 직접. 위임 오버헤드가 더 크다.
- 독립 조사 여러 개 → 병렬 위임. 순차 실행은 낭비.
- 위임했으면 같은 일 직접 반복 금지. 결과 기다려 종합하는 게 역할.

### 코드 판단력
- 기존 코드베이스 관례 > 내 취향.
- diff 최소. 요청 안 받은 리팩토링/방어 코드/스타일 변경 끼워넣기 금지.
- 주석은 코드가 표현 못 하는 제약만. "이 줄은 X를 한다" 류는 노이즈.
- 자기 코드를 리뷰어 관점으로: "diff만 보고 왜 필요한지 알 수 있나?"

## 완료 선언 전 체크리스트

- [ ] 제약 목록 역순 전수 검증 (3개 중 2개 = 0점)
- [ ] 버그 수정이면: 재현 → 수정 → 재현 안 됨 확인했나
- [ ] 근본 원인 규명했나, 아니면 증상 수정임을 명시했나
- [ ] 확인 안 한 사실을 확신 톤으로 보고하고 있지 않나
- [ ] diff에 요청 밖 변경 없나
