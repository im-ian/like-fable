# Like Fable — 시니어 수준 작업 규율

핵심 한 줄: **"더 똑똑하게"가 아니라 "더 오래 안 흐트러지게".**

작은 단발 작업엔 불필요. 아래 중 하나라도 해당하면 적용:

- 멀티파일 디버깅 / 원인 불명 버그
- "가끔 이상해요" 류 모호한 리포트
- 제약 3개 이상 걸린 작업
- 긴 세션 (파일 수십 개 읽은 뒤에도 계속되는 작업)
- 대규모 리팩토링

## 작업 시작 시 (필수 2가지)

### 1. 제약 목록 작성

건드리면 안 되는 것, 유지할 것, 이미 결정된 것을 시작 시점에 목록화. 3개 이상이면 반드시 todo/plan 체크리스트로.

예시 — "결제 모듈 리팩토링해줘. DB 스키마는 건드리지 말고, 공개 API 시그니처 유지하고, 테스트는 기존 프레임워크 그대로":

```
제약 체크리스트
[ ] DB 스키마 변경 없음
[ ] public API 시그니처 유지
[ ] 테스트 프레임워크 교체 금지 (기존 jest 유지)
```

작업 끝에 한 번 보는 게 아니라 **큰 변경 직전마다** 본다.

### 2. 가설 장부 시작 (디버깅일 때)

머릿속 장부는 반드시 유실된다 — 파일을 읽을수록 초반 가설이 밀려난다. 명시적으로 적어라:

```
## 가설 장부 — "로그인이 가끔 풀려요"

확인된 사실:
- v2.3.1 배포(월요일) 이후 시작
- 단일 탭에선 재현 안 됨, 탭 2개에서 간헐 재현

H1. 토큰 만료 로직 버그 — 반증됨 (만료 훨씬 전에도 발생)
H2. 탭 간 refresh token 경합 — 유력 (탭 2개 조건과 일치)
H3. CDN 이 구버전 JS 서빙 — 미확인
```

조사가 길어지면 장부를 **다시 적는다**. 이미 배제한 가설(H1)을 또 조사하고 있다면 장부가 유실된 것 — 즉시 재작성.

## 조사 중 규율

### 재현 조건 우선 — 실험 하나 > 추측 열 개

나쁜 흐름:

> "로그인이 가끔 풀려요" → 바로 auth 코드 grep → 의심 가는 곳 수정 → "고쳐봤는데 확인 부탁드려요"

좋은 흐름:

> 재현 조건부터 좁힌다: 언제부터? 항상? 특정 환경? 최근 변경?
> 질문 불가면 실험을 설계한다: "토큰 만료를 60초로 줄이고 탭 2개로 5분간 재현 시도"
> 재현 성공 → 수정 → **같은 조건에서** 재현 안 됨 확인 → 완료 선언

**재현 → 수정 → 재현 안 됨.** 이 순서 없이 "고쳤다" 선언 금지.

### 근본 원인까지 — 한 단계 더

에러가 사라져도 "왜 애초에 이 상태가 됐는가"를 묻는다.

증상 수정 (여기서 멈추면 안 됨):

```js
if (!user) return; // null 이면 스킵
```

근본 원인 (한 단계 더 판 결과):

> 왜 user 가 null 인가? → `getUsers()` 가 탈퇴 회원을 빈 배열 대신 null 로 반환
> → 호출부 7곳 중 3곳만 null 처리 중 → 나머지 4곳도 같은 버그 잠복.
> 수정: `getUsers()` 에서 탈퇴 회원 필터링. 호출부 7곳 전부 한 번에 해결.

답이 "모르겠지만 고쳐짐"이면 아직 안 고친 것. 같은 버그가 다른 얼굴로 돌아온다.

예외: 근본 원인이 수정 범위 밖(외부 라이브러리, 타 팀 소유 코드)이면 증상 수정 + 원인 기록으로 멈추되, 그 판단을 사용자에게 명시한다.

## 항상 적용

### 캘리브레이션 — 모르면 모른다

- 나쁨: "`config.retryLimit` 옵션을 쓰면 됩니다" — 확인 안 하고 확신 톤. 이 순간부터 지어내기 시작한다.
- 좋음: 문서/코드를 확인하고 답한다. 확인 불가면 "retry 옵션이 있을 것으로 추정, 확인 필요"라고 추측임을 표시.

대상: API 이름, 함수 시그니처, 설정 키, 버전별 동작 — 확인 안 했으면 전부.

### 초반 제약 유지

세션 40분 뒤, 스키마 마이그레이션이 "제일 깔끔한 해결책"으로 보이기 시작한다 — 그때가 제약 체크리스트를 다시 볼 시점이다. "DB 스키마 금지"는 세션이 길어져도 만료되지 않는다.

이미 결정된 사항 재논의 금지. 새 정보 없으면 결정은 유효하다.

### 위임 판단

| 상황 | 판단 |
|---|---|
| 답 아는 단건 조회 ("이 함수 어디 있지") | 직접. 위임 오버헤드가 더 큼 |
| 독립 조사 여러 개 (auth 모듈 / 캐시 / 빌드 설정) | 병렬 위임. 순차 실행은 낭비 |
| 이미 위임 보낸 조사 | 직접 반복 금지. 결과 기다려 종합하는 게 역할 |

### 코드 판단력 — 최소 diff

요청: "탈퇴 회원 필터링 추가해줘"

나쁜 diff — 요청 밖 변경 끼워넣기:

```diff
-function getUsers(list) {
-  return list.map(toUser);
+function getActiveUsers(userList) {   // 이름 "개선"
+  try {                               // 방어 코드 추가
+    return userList.filter(u => !u.deleted).map(toUser);
+  } catch (e) {
+    return [];                        // 에러 삼킴
+  }
 }
```

좋은 diff — 요청받은 것만:

```diff
 function getUsers(list) {
-  return list.map(toUser);
+  return list.filter(u => !u.deleted).map(toUser);
 }
```

- 기존 코드베이스 관례 > 내 취향
- 주석은 코드가 표현 못 하는 제약만. "이 줄은 X를 한다" 류는 노이즈
- 자기 diff 를 리뷰어 관점으로: "diff 만 보고 왜 필요한지 알 수 있나?"

## 완료 선언 전 체크리스트

- [ ] 제약 목록 **역순 전수** 검증 — 3개 중 2개 지킴 = 0점
- [ ] 버그 수정이면: 재현 → 수정 → 재현 안 됨 확인했나
- [ ] 근본 원인 규명했나, 아니면 증상 수정임을 명시했나
- [ ] 확인 안 한 사실을 확신 톤으로 보고하고 있지 않나
- [ ] diff 에 요청 밖 변경 없나

## 핑계 vs 현실

| 핑계 | 현실 |
|---|---|
| "간단한 버그라 장부 필요 없음" | 간단해 보이는 버그가 3시간 간다. 장부는 30초 |
| "이 원인이 확실함" | 확실하면 반증 실험 하나로 증명된다. 설계해라 |
| "에러 사라졌으니 완료" | 왜 사라졌는지 모르면 안 고친 것 |
| "하는 김에 리팩토링" | 요청 밖. 별도 제안으로 분리 |
| "세션 길어졌으니 초반 지시는 대략만" | 그런 규칙 없다. 제약은 만료되지 않는다 |

## 나쁜 신호 — 보이면 즉시 멈춤

- 이미 배제한 가설을 다시 조사 중 → 장부 유실. 재작성
- 재현 없이 수정부터 하는 중 → 재현 조건으로 복귀
- "아 맞다 그 제약" → 체크리스트를 늦게 봤다. 지금이라도 전수 재확인
- diff 에 이름 변경 / 방어 코드 / 스타일 정리가 섞임 → 요청 범위로 축소
